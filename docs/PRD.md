# KaoBot 产品需求文档 (PRD)

> 版本: v1.2 | 更新日期: 2026-03-30

---

## 1. 产品定位

**一句话描述**: KaoBot 是一个基于 AI Agent 的考研信息智能助手，通过高精度知识检索帮助考生获取导师信息、备考资料和个性化辅导。

**核心差异化**: 使用 PageIndex（推理式 RAG）替代传统向量检索，对结构化学术文档达到 95%+ 检索准确率。

---

## 2. 用户画像

| 维度 | 描述 |
|------|------|
| **目标用户** | 考研学生（大三/大四/二战/在职考研） |
| **核心痛点** | 导师信息分散、招生状态不透明、真题难获取、复试准备无方向 |
| **使用场景** | 择校阶段选导师、初试后查复试信息、复试前准备简历/自我介绍 |
| **技术水平** | 普通大学生，能用命令行或网页即可 |

### 3.2 F1: 导师信息检索（MVP 核心功能）

#### 用户故事

```
作为一名考研学生，
我想输入"清华大学计算机系做NLP方向的导师"，
系统能返回匹配导师列表，包含研究方向、是否招生、近期论文、联系方式，
以便我做出选导师决策。
```

#### 典型用户场景

**场景 A — 大三学生初次选校，广撒网找方向**

> 小陈是大三学生，刚决定考计算机专业研究生，对"NLP/大模型"感兴趣但不了解各校情况。
> 他打开 KaoBot，输入"北京地区 NLP 方向有哪些不错的导师"，系统返回多位导师卡片。
> 他追问"这几位导师今年都招生吗？哪位论文产出更多？"，通过多轮对话缩短了从"方向感兴趣"到"锁定目标导师"的决策时间，从原本需要一周翻论坛缩短为半小时。

**场景 B — 备考中期对比两位导师，做最终决策**

> 小王已备考 3 个月，初步选定了 A、B 两位导师，但不确定哪个更适合自己。
> 他上传了两位导师的个人主页截图和实验室招生公告，输入"帮我对比这两位导师的研究方向和招生要求"。
> KaoBot 生成对比表格，并根据他之前提到的"对知识图谱感兴趣"给出倾向性建议，帮他做出最终选择。

**场景 C — 成绩出来后紧急确认招生状态**

> 小李初试成绩刚出，估分 380，心仪导师是否今年招生成了最焦虑的问题。
> 他输入"XX 教授 2026 年还招硕士吗"，系统返回招生状态和置信度标注。
> 若状态不确定，系统主动提示"该信息来自去年招生简章，建议发邮件确认"，并给出联系方式，减少他的信息焦虑。

#### 功能需求

| 子功能 | 描述 | 数据来源 |
|--------|------|---------|
| 按姓名搜索导师 | 输入"张三 清华大学" → 返回导师完整信息 | 知识库（用户上传文档） |
| 按研究方向搜索 | 输入"NLP方向 北大" → 返回该方向所有导师 | 知识库 |
| 招生状态查询 | 当年是否招收硕士/博士 | 知识库 + 招生简章文档 |
| 导师论文查询 | 近5年代表性论文、研究趋势 | Semantic Scholar API |
| 导师对比 | 输入多位导师名 → 对比研究方向/论文产出 | 知识库 + API |
| 多轮对话 | 支持追问"他今年招几个？""他的学生就业去向？" | Agent 记忆 |

#### 数据来源策略（Phase 1: 手动上传优先）

```
Phase 1 (MVP):
  用户手动上传文档（PDF/Word/网页截图）
    → PageIndex 建立索引
    → Agent 通过推理式检索回答问题

Phase 2 (增强):
  + 爬虫自动采集院校官网数据
  + 公开 API 数据（Semantic Scholar 等）
  + 定期更新机制
```

#### 核心输出格式

```
== 导师信息卡片 ==
姓名: 张三 / San Zhang
职称: 教授 / 博导
院校: 清华大学 计算机科学与技术系
研究方向: 自然语言处理、大语言模型、知识图谱
招生状态: 2026年招收硕士 2名、博士 1名 [置信度: 高]
代表论文:
  1. "xxx" (ACL 2025, 被引 120)
  2. "xxx" (NeurIPS 2024, 被引 85)
联系方式: zhangsan@tsinghua.edu.cn
个人主页: https://xxx
数据来源: [清华CS官网] [Semantic Scholar]
最后更新: 2026-03-15
```

### 3.3 F2: 初试历年真题库

#### 用户故事

```
作为一名备考中的考研学生，
我想搜索"清华大学计算机专业2020-2025年初试真题"，
系统能返回完整真题文件及 AI 解析，
以便我了解出题规律、重点方向，高效备考。
```

#### 典型用户场景

**场景 A — 暑假刚开始备考，想摸清出题规律**

> 小张 7 月开始备考，专业课是"计算机组成原理"。她不知道哪些章节是高频考点，翻论坛找到的真题格式参差不齐。
> 她上传近 5 年真题 PDF，输入"帮我分析清华计算机初试专业课的高频考点分布"。
> KaoBot 列出各章节出题频率、题型占比，她据此制定了"先攻高频，后补弱项"的复习计划，节省了自己逐题整理的时间。

**场景 B — 距考试 2 个月，跨考生摸底**

> 小赵是跨考生，本科学经济，目标考软件工程。他不知道目标院校的专业课考哪些范围，网上信息互相矛盾。
> 他输入"浙大软件工程专业课 2021-2025 年初试题型分布"，KaoBot 给出结构化分析，并提示"数据结构占比约 40%，算法设计占 30%"。
> 他第一次有了清晰的复习方向，避免了跨考生最常见的"复习方向跑偏"问题。

**场景 C — 最后冲刺阶段，横向对比降低风险**

> 小孙在 A、B 两所院校之间犹豫，担心一所题目太难。
> 她输入"北大和人大数学难度对比，哪个更适合我的基础"，系统结合她上传的历年真题给出对比分析，帮她最终选择了更匹配自己水平的院校，降低了"高考志愿式失误"的风险。

#### 用户痛点（为什么愿意付费）

| 痛点 | 现状 | KaoBot 解法 |
|------|------|------------|
| 真题分散难找 | 各论坛碎片化，格式混乱 | 统一收录，结构化检索 |
| 不知道考什么重点 | 靠自己逐年翻看 | AI 自动归纳高频考点 |
| 题目无解析 | 只有裸题，不知道怎么答 | AI 生成参考解析 |
| 院校真题差异大 | 需逐个院校研究 | 跨院校横向对比 |

#### 功能需求

| 子功能 | 描述 | 数据来源 |
|--------|------|---------|
| 真题检索 | 按院校 / 专业 / 年份检索历年初试真题 | 用户上传 + 爬虫（Phase 2） |
| AI 考点分析 | 自动提取高频考点、题型分布 | PageIndex + LLM 分析 |
| 跨院校对比 | "清华和北大数学题难度对比" | 知识库 |
| 真题下载 | 导出 PDF | 文件存储 |

#### 付费意愿评估

- **意愿程度**: ⭐⭐⭐（中等）
- **付费场景**: 学生在初试前 3-6 个月进入高强度备考期，有明确的真题需求
- **竞品情况**: 小红书/知乎可以找到零散真题，但无 AI 解析；研招网仅有部分官方样题
- **定价建议**: 按院校包订阅（9.9-29.9元/所院校）或年费会员

### 3.4 F3: 复试笔试真题库

#### 用户故事

```
作为一名初试成绩刚过线、进入复试的学生，
我想搜索"浙江大学软件工程复试笔试近5年真题"，
系统能返回题目及考察重点，
以便我在2-4周内快速针对性备考，提高复试通过率。
```

#### 典型用户场景

**场景 A — 收到复试通知，不知从何下手（最典型场景）**

> 小林初试 390 分，收到复试通知后极度兴奋又极度焦虑——距复试只有 3 周，但不知道该复习什么。
> 他输入"浙大软件工程复试笔试考什么，历年题目有哪些规律"，系统返回近 4 年真题分析：操作系统占 40%，数据结构占 35%，网络占 25%。
> KaoBot 基于他的时间（3 周）自动生成了备考计划，让他从"不知道从哪开始"的焦虑状态进入有节奏的备考模式。

**场景 B — 想针对目标导师定制备考方向**

> 小吴已知道自己要面试的导师方向是"分布式系统"，担心考官的问题会偏向该方向。
> 他输入"XX 教授的研究方向，复试面试可能涉及哪些专业题目"，系统结合导师论文方向和历年复试笔试题，推测可能的考察侧重，帮他做针对性准备。
> 这种"精准备考"是传统资料无法实现的，极大提升了他的复试信心。

**场景 C — 时间极短，需要最高效的刷题方案**

> 小陈复试通知下来时距面试只剩 12 天，根本没时间系统复习。
> 她输入"中科院计算所复试笔试，12 天高频考点速刷方案"，系统生成精简版备考计划，只覆盖历年真题中出现频率 ≥ 2 次的核心知识点。
> 针对"时间换成功率"的极端场景，KaoBot 提供了最优的时间分配策略。

#### 用户痛点（为什么愿意付费）

| 痛点 | 现状 | KaoBot 解法 |
|------|------|------------|
| 复试真题极难找 | 比初试更稀缺，各院校差异极大 | 系统化收录 |
| 时间窗口极短 | 复试备考只有 2-4 周 | 快速精准检索 |
| 不知道考哪些方向 | 导师方向不同，考察侧重不同 | 按导师方向匹配考题 |
| 焦虑状态下决策困难 | 过线了怕浪费机会，急需信息 | 结构化引导备考 |

#### 功能需求

| 子功能 | 描述 | 数据来源 |
|--------|------|---------|
| 复试真题检索 | 按院校 / 专业 / 导师方向检索复试笔试题 | 用户上传 + 社区贡献 |
| 考察重点分析 | "这位导师复试偏爱考什么" | 知识库 + LLM 分析 |
| 备考计划生成 | 基于已有时间自动生成复习计划 | LLM 推理 |
| 模拟题生成 | 按已有真题风格生成模拟练习题 | LLM 生成 |

#### 付费意愿评估

- **意愿程度**: ⭐⭐⭐⭐⭐（最高，与 F4/F5 并列）
- **付费场景**: 初试过线 → 复试通知下来 → 极度焦虑、时间极短，愿意花钱换时间和确定性
- **关键洞察**: 此时用户的心态是"我已经投入了大半年，不能在最后一步失败"，付费意愿最强
- **定价建议**: 按次（29.9-49.9元/院校）或复试专项包（99-199元，含 F3+F4+F5）

### 3.5 F4: 复试简历定制

#### 用户故事

```
作为一名进入复试的学生，
我想输入自己的本科背景、项目经历和目标导师信息，
系统能生成一份针对该导师研究方向的个性化简历，
以便在复试面试中给导师留下深刻印象，提高录取概率。
```

#### 典型用户场景

**场景 A — 背景普通，想靠简历"以弱胜强"**

> 小王本科 211，GPA 3.2，没有论文，目标导师是做"机器学习"的，竞争者中不乏 985 背景的同学。
> 他填入自己的两段实习和一个课程项目，系统分析导师近期论文后，发现其中一段实习与导师研究课题高度相关。
> KaoBot 生成的简历重点突出了这段经历，并用导师论文中的技术词汇描述项目内容，让他的简历在同类竞争者中显得更有针对性。

**场景 B — 有科研经历，不知道如何专业化表达**

> 小赵有一篇在投论文，但不知道如何在简历中体现其学术价值，描述时总是"我做了一个 XX 系统"这样平白的语言。
> KaoBot 引导她按"研究问题 → 方法创新点 → 实验结果"的框架重新表达，并对齐目标导师的研究语境。
> 修改后的描述立刻从"项目式"升级为"科研式"，大幅提升了专业感。

**场景 C — 同时准备多所院校，需要不同版本**

> 小孙进了两所学校的复试，导师方向一个偏 CV，一个偏 NLP，通用简历无法突出匹配点。
> 他一次录入背景信息后，选择"生成 CV 版"和"NLP 版"，系统自动调整各版本中的课程侧重和项目描述角度。
> 一次录入两份成品，为他节省了大量重复修改的时间，同时保证每份简历都有针对性。

#### 用户痛点（为什么愿意付费）

| 痛点 | 现状 | KaoBot 解法 |
|------|------|------------|
| 不知道如何突出与导师的匹配点 | 只会写通用简历 | 基于导师方向定制，突出相关经历 |
| 担心简历显得外行 | 不懂如何描述科研经历 | AI 帮助专业化表达 |
| 时间太短 | 复试备考已占满时间 | 5 分钟完成定制 |
| 不知道导师看重什么 | 信息不透明 | 结合导师论文方向分析偏好 |

#### 功能需求

| 子功能 | 描述 | 数据来源 |
|--------|------|---------|
| 背景信息录入 | 引导式填写：本科院校/GPA/项目/论文/竞赛 | 用户输入 |
| 导师匹配分析 | 分析用户背景与目标导师研究方向的契合点 | 知识库 + LLM |
| 简历生成 | 生成 PDF/Word 格式定制简历 | LLM 生成 + 模板 |
| 多轮优化 | 支持用户反馈，多轮迭代修改 | Agent 多轮对话 |
| 多导师版本 | 一次录入，生成多份不同侧重的简历 | LLM 批量生成 |

#### 核心输出格式

```
== 复试简历（针对 XX 导师定制版）==

[头部]
姓名 | 本科院校 | 专业 | GPA
联系方式 | GitHub | 个人主页

[为什么选择 XX 导师 — 匹配亮点]
您在 [研究方向A] 的工作启发了我在 [相关项目] 中的思路...

[教育背景]
...（突出与导师方向相关的课程）

[科研/项目经历]（按与导师方向相关度排序）
项目名 | 角色 | 时间
- 做了什么（技术词汇与导师论文语言对齐）
- 取得了什么结果

[论文/竞赛]（如有）
...
```

#### 付费意愿评估

- **意愿程度**: ⭐⭐⭐⭐⭐（最高）
- **付费场景**: 高度个性化 + 时间紧迫 + 高焦虑 = 最强付费驱动
- **定价建议**: 单次生成（19.9元/份）或复试包（F3+F4+F5 打包 99-199元）

### 3.6 F5: 面试自我介绍定制

#### 用户故事

```
作为一名即将参加复试面试的学生，
我想输入我的背景和目标导师信息，
系统能生成1-3分钟的中英文自我介绍脚本，并提供练习反馈，
以便我在面试开始时建立良好的第一印象。
```

#### 典型用户场景

**场景 A — 第一次参加研究生面试，完全不知道说什么**

> 小陈从没参加过正式学术面试，对着"自我介绍"这四个字完全不知如何开口。
> 她填入基本信息后，KaoBot 生成了 1 分钟和 3 分钟两个版本，自然地串联了"为什么读研 → 为什么选这个方向 → 自己的相关经历 → 期望与导师的合作方向"。
> 她反复练习后，从"脑子一片空白"变成了"有逻辑、有重点"，进入考场时焦虑感明显降低。

**场景 B — 需要英文自我介绍，口语底子薄**

> 小李目标院校复试有英文面试环节，他的口语只是日常水平，担心学术表达不专业。
> 他让 KaoBot 生成英文版本，系统不仅措辞地道，还标注了"distributed systems""few-shot learning"等关键术语的发音要点。
> 他按照脚本练习了 5 遍，面试时顺利完成了英文自我介绍，考官也给予了正面反馈。

**场景 C — 多轮打磨，精细调整到令自己满意**

> 小孙收到生成的脚本后，感觉"科研经历部分说得太长，口语感不强"。
> 她对 KaoBot 说"把第二段科研部分压缩到 30 秒，语气更口语化一些"，系统根据反馈修改。
> 经过 3 轮对话式打磨，最终版本完全符合她的表达习惯，而不是生硬地照读 AI 生成的文本。

#### 用户痛点（为什么愿意付费）

| 痛点 | 现状 | KaoBot 解法 |
|------|------|------------|
| 不知道该说什么 | 只会背通用模板，显得生硬 | 结合导师兴趣定制，有针对性 |
| 英文自我介绍更难 | 担心口语表达不专业 | AI 生成地道学术英文脚本 |
| 时间控制不好 | 说太长/太短都影响印象 | 提供 1min / 2min / 3min 版本 |
| 临场紧张 | 没有充分练习 | 模拟问答练习 |

#### 功能需求

| 子功能 | 描述 |
|--------|------|
| 脚本生成 | 生成中文 + 英文自我介绍，含时间版本（1/2/3分钟） |

- **付费场景**: 通常与 F4 简历捆绑使用，打包付费意愿更强
- **定价建议**: 与 F4 打包，单独购买 14.9元

#### 典型用户场景

**场景 A — 大三择校，不知道报哪个档次**

> 小吴大三，预估自己初试能考 360 左右，想报计算机专业，但不确定该冲 985 还是选稳妥的 211。
> 他输入"计算机 NLP 方向，预计分数 355-370，本科 985 但 GPA 一般，推荐怎么选"，系统返回冲/稳/保三档院校推荐，附上近 3 年各校录取分数线趋势。
> 有了数据支撑的选择，他从"要不要冲清华"的迷茫中解脱出来，把精力集中在备考上。

**场景 B — 调剂阶段，成绩过线但落榜，5 天内找出路**

> 小刘初试 330 分，刚过国家线，第一志愿未进复试。调剂系统刚开放，他看到海量院校信息却不知如何排优先级，极度焦虑。
> 他输入"我 330 分，软件工程，本科 211，还有哪些院校在接受调剂"，系统实时筛选出仍有名额、截止时间在 3 天以上的院校，并按他的分数/背景自动标注成功率。
> 5 天的黄金窗口内，他没有在信息筛选上浪费时间，而是把精力集中在准备调剂材料上，最终成功上岸。

**场景 C — 调剂最后 48 小时，紧急批量发邮件**

> 小程调剂截止剩 2 天，手头有 6 所院校联系方式，需要立刻发出调剂意向邮件，但不知道如何写才能显得真诚且有竞争力。
> 他输入"帮我生成发给 XX 大学的调剂意向邮件，我的情况是……"，系统生成专业的邮件模板，突出他与该院校/导师方向的匹配点。
> 6 封个性化邮件在 30 分钟内发出，而不是用同一封通用模板批量群发，提升了回复率。

### 5.2 OpenClaw 生态集成规划

KaoBot 设计上与 OpenClaw 生态对齐，后续将深度集成。

#### 为什么选择 OpenClaw

| 维度 | OpenClaw 提供的能力 | KaoBot 受益点 |
|------|-------------------|--------------|
| **多通道接入** | Slack / Discord / WhatsApp / Web / CLI 统一接入 | 考生可通过微信/Slack/Web 等多渠道使用 KaoBot |
| **会话管理** | 持久化会话、跨通道上下文保持 | 复试准备全周期对话不丢失 |
| **Skills 系统** | 插件化能力注册（SKILL.md 声明式配置） | 各功能模块(F1-F6)独立注册为 Skill |
| **Heartbeat 调度** | Cron 任务、定时触发 | 定时更新导师招生状态、提醒考生重要时间节点 |
| **Memory 系统** | Markdown 文件持久化记忆 | 考生画像、目标院校、备考进度长期跟踪 |
| **安全沙箱** | 工具执行隔离、权限控制 | 爬虫/API 调用安全管控 |

#### 集成策略（渐进式）

```
Phase 1 (MVP): KaoBot 独立运行
  - 自建轻量 Gateway + ReAct Brain
  - CLI 交互
  - 架构设计兼容 OpenClaw 接口规范

Phase 2: 接入 OpenClaw Gateway
  - KaoBot Agent 注册为 OpenClaw Skill
  - 通过 OpenClaw 多通道接入（Web/Slack/Discord）
  - 使用 OpenClaw 会话管理替代自建 Memory

Phase 3: 深度集成
  - 多 Agent 协作: TutorAgent + ExamAgent + ResumeAgent 通过 OpenClaw 编排
  - Heartbeat 定时任务: 招生信息自动更新、考试时间提醒
  - sessions_spawn: Agent 间任务委派（如 TutorAgent 委派 ScholarAgent 查论文）
```

#### KaoBot Skill 注册示例（Phase 2）

```
# skills/kaobot-tutor/SKILL.md
---
name: kaobot-tutor
description: 考研导师信息检索 - 查询导师研究方向、招生状态、论文、联系方式
triggers:
  - "导师"
  - "教授"
  - "招生"
  - "研究方向"
  - "tutor"
  - "supervisor"
---

你是一个考研导师信息助手。当用户询问导师相关问题时，
使用 PageIndex 知识库检索和 Semantic Scholar API 获取信息，
以"导师信息卡片"格式返回结果。
```

#### 架构兼容性设计原则

1. **Skill-Ready**: 每个 Agent 模块独立，可直接映射为 OpenClaw Skill
2. **Memory 兼容**: 长期记忆使用 Markdown 文件格式（与 OpenClaw 一致）
3. **Tool 标准化**: 工具接口遵循 OpenAI Function Calling 规范（OpenClaw Brain 原生支持）
4. **MCP 协议**: PageIndex 通过 MCP 接入，OpenClaw 可直接复用

## 6. MVP 定义与交付标准

### MVP 范围（Phase 1 交付物）

1. **文档上传与索引**: 用户通过 CLI 上传 PDF/文档 → PageIndex 自动建立索引
2. **导师信息问答**: 基于已索引文档的多轮对话问答
3. **论文检索增强**: 自动补充 Semantic Scholar 论文数据
4. **CLI 交互**: 终端交互式聊天 + 直接命令

### MVP 验收标准

| 编号 | 验收项 | 标准 |
|------|--------|------|
| AC-1 | 文档上传 | `kaobot ingest xxx.pdf` 成功建立 PageIndex 索引 |
| AC-2 | 基础问答 | 针对已上传文档提问，回答准确率 >= 95%（20题人工评测） |
| AC-3 | 导师搜索 | "XXX教授的研究方向是什么" 返回正确结果 + 来源引用 |
| AC-4 | 招生查询 | "XXX今年是否招生" 返回状态 + 置信度标注 |
| AC-5 | 多轮对话 | 支持至少3轮上下文追问 |
| AC-6 | 论文补充 | 自动从 Semantic Scholar 补充导师论文信息 |
| AC-7 | 错误处理 | LLM离线/PageIndex不可用时有明确提示 |

### 不在 MVP 范围内

- Web UI（Phase 2）
- 自动爬虫采集（Phase 2，目前作为 TODO）
- 简历生成、面试准备、调剂推荐（Phase 2+）
- 多用户/账号系统
- 数据自动更新
