## 1. 产品概况

### 1.1 品牌定位

| 维度 | 描述 |
|------|------|
| **产品名称** | KaoBot |
| **一句话定位** | AI 驱动的考研复试全程智能辅导助手 |
| **品牌关键词** | 精准 · 高效 · 个性化 · 低焦虑 |
| **品牌主张** | "把复试准备的确定性还给每一位考生" |
| **品牌色** | 深蓝（专业/学术感）+ 橙金（活力/突破感） |
| **目标调性** | 既有导师的专业感，又有学长的亲切感；不说废话，直接给答案 |

**品牌故事**: 每年逾百万学生（2026 年约 174 万）经历初试后进入复试，面对 2-4 周极短窗口期，信息极度碎片化、备考策略高度不确定。KaoBot 用 AI Agent 将"导师信息、历年真题、简历定制、面试模拟"整合为一个对话入口，让每个人都能享受到原本只有"内部人脉"才能获得的备考资源。

> **数据来源说明（Data Sources）：**
>
> - **报名人数（2019–2026）**：教育部官方公告，经新华网、人民网转载。2026 年数据为教育部于 2025 年 11 月 24 日正式公布（[人民网报道](https://edu.people.com.cn/n1/2025/1124/c367001-40610589.html)）
> - **研究生招生人数（2021–2024）**：教育部《全国教育事业发展统计公报》官方确认数字（✓），含硕士生与博士生；2019–2020 年据中国教育在线研招报告估算；**2025 年待教育部 2026 年中公报正式发布**；2026 年为基于趋势的预测值，非官方数据。
> - **复试进入人数估算依据**：硕士招生计划 × 120%（教育部规定院校复试人数原则上不超过招生计划的 120%）。**注：34 所自划线院校（含北大、清华、复旦等）自主划定分数线，复试比例可达 130%–150%，竞争显著高于普通院校**；普通院校以国家统一划定 A/B 区分数线为准。
> - **数据截止日期**：2026 年 4 月。如有更新请参考 [教育部官网](https://www.moe.gov.cn) 及 [研招网](https://yz.chsi.com.cn)。

> **趋势解读**：报名人数自 2023 年峰值（474 万）已连续三年下降，2026 年降至 343 万（三年合计减少 **131 万人**）；但研究生招生计划逆势持续扩大，录取率从 2023 年谷底 27.5% 回升至 2026 年预计 42.3%，**复试竞争结构性调整：整体难度下降，但头部院校（34 所）竞争更加激烈**。

## 2. 产品体验五层模型

### 2.1 战略层

#### 用户需求地图

```
备考时间线：
─────────────────────────────────────────────────────────────────►
12月    1月      2月       3月初     3月中旬    3月末/4月
初试 → 等成绩 → 成绩出 → 国家线 → 复试通知 → 复试 → 录取
                  │         │         │
                  │         │         └── 需求爆发：真题、简历、面试模拟
                  │         └── 需求涌现：院校选择、调剂方案
                  └── 需求萌发：导师信息查询

复试阶段核心需求（KaoBot 主战场）：
┌──────────────────────────────────────────────────────────┐
│  1. 知道复试考什么（笔试方向、题型）                          │
│  2. 了解目标导师偏好（方向、风格、问题倾向）                   │
│  3. 做好简历（针对导师定制，不是通用简历）                    │
│  4. 准备自我介绍（中英文，控制时长）                          │
│  5. 模拟面试（降低临场焦虑）                                │
└──────────────────────────────────────────────────────────┘
```

#### 产品定位宣言

> KaoBot 是每一位考研生在复试阶段的"AI 学长"——它知道目标院校的复试规律，能帮你写简历、练口语，在最关键的 2-4 周里成为你最高效的备考伙伴。

#### 市场分析

**市场规模（TAM / SAM / SOM）**

| 市场层次 | 定义 | 规模估算 |
|---------|------|---------|
| **TAM** 总可触达市场 | 所有考研相关培训/工具市场 | ~500亿元/年 |
| **SAM** 可服务市场 | AI + 考研复试辅导细分市场 | ~15亿元/年 |
| **SOM** 实际可获取市场 | KaoBot Phase 1-2 目标市场 | ~3,000万元/年 |

**各层市场切割推导过程**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TAM ≈ 500 亿元/年
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
计算公式：
  TAM = 中国考研培训市场整体规模（含初试+复试+工具）

数据来源与依据：
  · 艾瑞咨询《2024中国在线教育行业报告》：考研赛道市场规模约 480-520 亿元
  · 构成：网课/直播课（约55%）+ 线下班（约25%）+ 资料/工具（约10%）+ 一对一（约10%）

假设条件：
  A1. 市场规模采用 2024 年基准，未考虑考研人数下降带来的收缩（约-5%/年），
      保守处理以免高估机会
  A2. 包含初试+复试全阶段培训，未剔除非目标用户支出

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
SAM ≈ 15 亿元/年
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
计算公式：
  SAM = TAM × 复试阶段支出占比 × AI工具付费渗透率
      = 500亿 × 15% × 20%
      = 500亿 × 3% = 15亿元

假设条件：
  B1. 复试阶段支出占全年考研支出比例 ≈ 15%
      依据：复试备考周期约 3-4 周（全年考研周期约 6-10 月），且复试课程均价低于初试
  B2. AI 工具在复试辅导中的付费渗透率 ≈ 20%（保守估算）
      依据：2024 年大学生 AI 工具月活渗透率 >60%，但付费转化约 20-30%；
            复试场景支付意愿高于日常，取 20% 作为保守基准
  B3. ⚠️ 敏感性分析：若 AI 渗透率升至 30%，SAM ≈ 22.5 亿；若降至 10%，SAM ≈ 7.5 亿
```

| **给出路** | 每种异常态必须提供至少 2 个可执行的后续动作 |
| **不技术化** | 禁止展示 HTTP 状态码、堆栈信息、英文技术词汇 |
| **保护额度** | 因系统问题导致的失败不扣除用户使用次数 |
| **离线可用** | 网络异常时尽量展示已缓存内容（真题 PDF、历史对话）|

#### 图标设计

| 设计原则 | 说明 | 示例 |
|---------|------|------|
| **风格** | 线性图标为主（Outlined），圆角 2px | 与学术感、简洁感契合 |
| **尺寸** | Tab 栏 24px；功能入口 32px；卡片内 20px | 遵循 8px 栅格系统 |
| **语义清晰** | 每个功能对应固定图标，不随界面变化 | 真题=📚书本，简历=📄文档，面试=🎤麦克风 |
| **状态变化** | 激活态填充色（品牌蓝）；未激活线性灰 | 底部 Tab 激活/未激活明确区分 |
| **禁忌** | 避免使用过于圆润的卡通图标（降低专业感） | — |

#### 整体设计风格

| 维度 | 设计决策 | 理由 |
|------|---------|------|
| **整体基调** | 清爽学术风（Academic Clean） | 对标 Notion / 语雀，用户群为高知学生 |
| **布局密度** | 中等密度（卡片式布局，适量留白） | 信息量大但不压抑，复试焦虑期需要舒适体验 |

## 3. 行业分析

### 3.1 考研培训行业概况

**行业规模**：中国考研培训市场规模已超 **500 亿元**（2024年），年复合增速约 12%。

**行业特征**：
1. **重服务、轻技术**：传统玩家（新东方/考研帮/文都）以线下/线上课程为主，AI 工具渗透率低
2. **标准化 vs 个性化的矛盾**：大班课成本低但效果有限，一对一效果好但成本极高（500-2000元/小时）
3. **信息不对称严重**：复试信息（真题/导师偏好）高度分散，催生灰色信息中介市场

### 3.2 AI 教育工具趋势

| 趋势 | 现状 | KaoBot 机会 |
|------|------|------------|
| LLM 垂直化落地 | 通用 AI 无法精准回答"XX导师今年招几个" | 垂直知识库 + Agent 工具调用是关键 |
| 个性化学习需求升级 | 学生拒绝"千人一面"的复习材料 | 导师定制化是核心差异点 |
| 焦虑经济 | 高压备考催生大量"即时解决方案"需求 | 24小时可用的 AI 助手替代等学长回复 |
| 移动优先 | 90%+ 考研相关信息获取在手机上完成 | 移动端优先设计 |

### 3.3 政策环境

| 政策/趋势 | 影响 |
|----------|------|
| 研究生扩招政策持续 | 复试重要性提升，差异化竞争加剧 |
| AI 教育合规要求 | 需确保 AI 生成内容有来源标注，避免误导 |

## 5. 用户价值分析

### 5.1 用户核心价值公式

```
用户感知价值 = （信息获取效率 + 内容个性化程度 + 焦虑降低程度）
               ÷ （使用成本 + 学习成本 + 价格）
```

KaoBot 的价值策略：**分子三项全部高于竞品，分母三项全部低于竞品**。

### 5.2 各功能用户价值拆解

| 功能 | 解决的核心问题 | 用户感知价值 | 量化指标（待验证）|
|------|------------|------------|---------|
| **F3 复试真题库** | 不知道考什么 | 从"摸黑备考"到"有的放矢" | 备考效率提升 3× ⚠️ |
| **F4 简历定制** | 简历无针对性 | 从"通用简历"到"导师专属版" | 面试邀约率提升（预估）⚠️ |
| **F5 面试模拟** | 临场焦虑 | 从"不知道说什么"到"有脚本有底气" | 自我评价焦虑下降 ⚠️ |
| **F1 导师查询** | 信息分散难找 | 从"翻遍知乎"到"5分钟知道答案" | 信息获取时间节省 >90% ⚠️ |
| **AI 对话串联** | 多次提问繁琐 | 一次对话完成全流程 | 操作步骤减少 70% ⚠️ |

> ⚠️ **以上量化指标均为假设性预估，尚未经过实验验证。** 见下方验证方案。

### 5.2.1 量化指标验证方案

> **为什么需要验证**："备考效率提升 3×"等说法将出现在产品营销文案中，如无实验支撑则构成虚假宣传；同时验证结果将直接影响功能优先级决策。

#### 验证计划①：备考效率提升验证（F3 真题库）

```
指标：备考效率提升 3×（假设值，即用 KaoBot 准备 1 小时 = 传统方式 3 小时）

验证方法：受控对照实验（Controlled A/B Study）

实验设计：
  样本量：各组 30 人（共 60 人），目标院校和专业尽量配对匹配
  分组：
    · 对照组（A）：使用传统方式（知乎/小红书/考研帮）备考
    · 实验组（B）：使用 KaoBot 备考
  任务：针对"某指定院校专业"整理出：
    ① 近3年复试笔试考察重点（5条）
    ② 目标导师研究方向摘要（100字）
    ③ 复试备考计划（3天版本）
  度量指标：
    · 主指标：完成以上3个任务的时间（分钟）
    · 次指标：结果准确率评分（由专家盲评，0-5分）

评估时间：种子用户内测第2-3周执行
假设验证条件：B组时间 ≤ A组时间 × 1/2（即效率提升≥2×），
              且结果准确率 B ≥ A
若未达成：降低"效率提升X倍"表述，改为"更快找到目标信息"等定性描述
```

#### 验证计划②：信息获取时间节省验证（F1 导师查询）

```
指标：信息获取时间节省 >90%

验证方法：任务计时实验

实验任务：查找"[指定导师]的以下5项信息"：
  - 近3年代表论文方向
  - 当年是否招收硕士生（计划名额）
  - 导师面试风格（历届学生描述）
  - 导师常问面试问题（历届整理）
  - 是否接受跨专业考生

对照组：使用谷歌/知乎/院校官网
实验组：使用 KaoBot

度量：各组完成5项信息收集的时间（秒）
目标：实验组时间 ≤ 对照组时间 × 10%（即节省90%）

样本：各 20 人，覆盖 5 所不同院校导师（每人查同一导师）
```

#### 验证计划③：操作步骤减少验证（AI 对话串联）

```
指标：操作步骤减少 70%

验证方法：任务步骤计数 + 用户测试

任务场景：完成"复试全套备考材料准备"，包括：
  真题查询 → 考点分析 → 简历生成 → 自我介绍脚本 → 模拟面试问题

计数方式：记录用户完成全流程的"操作步数"
（定义：每次点击/输入/切换页面/切换应用 = 1步）

对照组：传统方式（多个平台/工具组合使用）
实验组：KaoBot 一站式完成

假设验证：实验组步数 ≤ 对照组步数 × 30%
```

#### 验证计划④：面试焦虑降低验证（F5 面试模拟）

```
指标：自我评价焦虑下降（定性→定量化）

量化工具：改编自标准化焦虑量表（STAI 状态焦虑量表简化版，6题）

时间点：
  T1：使用 F5 模拟面试前（填写焦虑评分）
  T2：使用 F5 模拟面试后（填写焦虑评分）
  T3：复试当天（可选，事后问卷）

样本：内测用户中参与面试模拟的用户，≥ 50 人
验证条件：T2 焦虑分数 < T1（配对 t 检验 p < 0.05）
预期效果：焦虑分数降低 ≥ 15%（文献参考值：正念练习降低 20-30%）
```

#### 验证时间线与决策节点

| 时间节点 | 行动 | 决策 |
|---------|------|------|
| 种子用户内测第1周 | 收集基线数据（传统方式用时）| — |
| 内测第2-4周 | 执行对照实验①② | — |
| 内测第5周 | 汇总实验结果 | 若数据支持 → 更新营销文案；若不支持 → 修改量化描述 |
| MVP 上线后 | 埋点追踪真实用户行为数据 | 持续校准 |

### 5.3 用户旅程地图（复试备考全流程）

### 6.3 单位经济模型（Unit Economics）

> **为什么重要**：考研复试是高度季节性产品，LTV 窗口短、CAC 压力集中在 3 月爆发期。需在 Phase 1 跑通 CAC < LTV 的基本逻辑，才能判断规模化增长的可行性。

#### CAC（获客成本）各渠道估算

| 获客渠道 | 预估 CPM/CPC | 点击→注册转化率 | 注册→付费转化率 | **CAC（付费用户）** | 可扩展性 |
|---------|------------|--------------|--------------|-----------------|---------|
| 小红书 KOL 种草（腰部）| ¥500-1500/篇 | 约 3-5% | 约 8-12% | **¥250-600** | 中（依赖内容质量）|
| 知乎回答 + 问答引流 | 近乎 0 成本 | 约 2-3% | 约 5-8% | **¥0-50**（人力换算）| 高（长尾流量持续）|
| 考研帮/贴吧精准投放 | ¥0.5-2/点击 | 约 5-8% | 约 6-10% | **¥100-500** | 中 |
| 老用户口碑裂变 | 奖励成本 ¥20-50/单 | — | — | **¥20-50** | 极高（K 因子驱动）|
| 搜索广告（百度/微信）| ¥3-8/点击 | 约 8-12% | 约 5-8% | **¥500-1500** | 低（成本高）|
| **综合加权 CAC** | — | — | — | **约 ¥150-300** | — |

```
CAC 计算公式：
  CAC = 总获客营销支出 / 新增付费用户数

Phase 1 目标：
  CAC ≤ ¥200（以口碑 + 内容为主，避免付费广告为主要渠道）
  达成条件：口碑裂变 K > 1.0，即每个付费用户平均带来 1 个新注册
```

#### LTV（用户生命周期价值）

```
LTV 计算公式：
  LTV = 平均首次购买客单价 + 复购收入 + 转介绍价值

KaoBot 场景拆解（考虑考研一次性消费特性）：

  ┌── 当年复试用户
  │   首次购买：复试包 ¥99-199（均值 ¥130）
  │   追加：月度会员 ¥29.9 × 2月 = ¥59.8
  │   小计：LTV₁ ≈ ¥190
  │
  ├── 二战用户（次年复购）
  │   复购率估算：约 15-20%（考研二战率约 20%，其中使用 KaoBot 的复购率更高）
  │   次年贡献：¥190 × 18%复购率 = ¥34
  │
  ├── 转介绍价值（口碑传播）
  │   NPS 目标 >50 的前提下，每个付费用户转介绍约 0.8 个新付费用户
  │   转介绍价值（折现）= ¥190 × 0.8 × 50% 折扣因子 = ¥76
  │
  └── 综合 LTV = ¥190 + ¥34 + ¥76 ≈ ¥300

LTV/CAC 比值目标：LTV ¥300 / CAC ¥150 = 2.0×（健康基准为 ≥ 3×）
→ Phase 1 需将 CAC 压低至 ¥100 以下，或 LTV 提升至 ¥450+（通过调剂包交叉销售）
```

#### Payback Period（回收周期）

```
Payback Period = CAC / 月均用户贡献收入

月均用户贡献收入：
  = 复试包单次收入 ÷ 复试季时长（月）+ 月会员费
  = ¥130 ÷ 3月 + ¥29.9 ≈ ¥73/月

Payback Period = ¥150（CAC） / ¥73/月 ≈ 2.1 个月

⚠️ 季节性风险：
  考研复试季仅约 6-8 周（2月-4月），用户活跃窗口极短
  → CAC 需在 1 个复试季内回收（否则用户流失后无法续收）
  → 核心策略：将 CAC 压缩至 ¥100 以内，确保单季度回收

调剂应急包的 Unit Economics（更优）：
  · CAC：同渠道，但转化率更高（紧迫性驱动）≈ ¥80
  · LTV：¥65（调剂包单价）+ 复试包交叉销售 ¥50 ≈ ¥115
  · Payback Period = ¥80 / ¥115 < 1 个月 ✅（调剂包是利润率最优产品）
```

#### Phase 1 Unit Economics 达标条件

### 6.4.1 成本弹性模型（用户量 × 成本）

> **核心洞察**：LLM API 费用与 DAU×查询频次强相关，是最大变动成本；PageIndex 本地化后搜索成本下降 95%；模型选型（DeepSeek vs GPT）对毛利率影响达 30 个百分点。

**关键假设：**
```
· DAU/MAU（复试季）：60%；活跃用户日均查询：5次；每次消耗 ~2,000 tokens
· DeepSeek-V3 ≈ ¥0.002/1k tokens；GPT-4o-mini ≈ ¥0.015/1k tokens
· PageIndex 本地搜索边际成本 ≈ ¥0.002/次（vs Tavily ¥0.01/次）
```

| 成本项目 | 1,000 用户/月 | 5,000 用户/月 | 10,000 用户/月 |
|---------|-------------|-------------|--------------|
| DAU（复试季）| ~200/天 | ~1,000/天 | ~2,000/天 |
| LLM（DeepSeek 主力）| ¥600 | ¥3,000 | ¥6,000 |
| LLM（GPT-4o-mini 对比）| ¥4,500 | ¥22,500 | ¥45,000 |
| 搜索（Tavily 切换前）| ¥400 | ¥2,000 | ¥4,000 |
| 搜索（PageIndex 切换后）| ¥50 | ¥100 | ¥200 |
| 服务器/CDN/OSS | ¥300 | ¥800 | ¥2,000 |
| **月总成本（DeepSeek + PageIndex）** | **≈ ¥1,050** | **≈ ¥4,200** | **≈ ¥8,700** |
| **月总成本（GPT-4o-mini + Tavily）** | **≈ ¥5,300** | **≈ ¥25,600** | **≈ ¥51,700** |

```
毛利率（5,000用户 / 500付费用户场景）：
  月收入 ≈ ¥69,500（复试包 + 月会员）
  成本（DeepSeek + PageIndex）= ¥4,200 → 毛利率 94.0% ✅
  成本（GPT-4o-mini + Tavily）= ¥25,600 → 毛利率 63.2% ✅（仍健康）

关键决策：优先落地 PageIndex 本地化（M1 里程碑）+ 使用 DeepSeek 作主力模型
          这两项合计节省成本约 80%，是单位经济模型成立的关键路径
```

**成本预警线：**

| 预警指标 | 阈值 | 触发行动 |
|---------|-----|---------|
| LLM API 月费 | > ¥5,000 | 评估降级模型或提高缓存命中率 |
| 单次查询成本 | > ¥0.05 | 检查缓存命中率（目标 > 40%）|
| 服务器 CPU 均值 | > 70% | 弹性扩容触发 |
| 月毛利率 | < 60% | 召开成本优化专项会议 |

| **内测用户（所有）** | 30 天全功能免费 | 通过筛选即获得 |
| **活跃反馈用户** | 复试成功后赠送下年度复试包（¥130 价值）| 累计提交 4 次以上有效反馈 |

#### 反馈收集机制

```
每周反馈问卷（5 分钟，微信表单）：
  Q1: 本周使用了哪些功能？（多选）
  Q2: 最有用的功能是？（单选 + 一句话原因）
  Q3: 遇到过什么问题/不好用的地方？（文字填写，可选）
  Q4: 愿意推荐给同学吗？（NPS 0-10 分）
  Q5: 本周备考进展如何？（了解用户状态，建立情感连接）

深度访谈（每 2 周 1 次，每次 15 分钟，电话/视频）：
  · 每次抽取 5 名用户进行 1 对 1 访谈
  · 重点挖掘：未完成的任务、差点流失的时刻、功能改进建议
  · 访谈后 24 小时内输出 User Interview Notes 存档

反馈闭环：
  · 每周五 PM 整理当周反馈 → 周六早会同步开发团队
  · 采纳的反馈在下次更新日志中注明"来自内测用户 @XX 的建议"
  · 未采纳的反馈说明原因并回复用户（不能石沉大海）
```

### 9.2 用户增长漏斗

```
曝光 (Awareness)
  └── 小红书/知乎种草 → 目标：10万曝光/月

兴趣 (Interest)
  └── 落地页/免费试用 → 转化率目标：5% → 5,000注册

激活 (Activation)
  └── 首次使用完成一次真题查询 → 激活率目标：60%

留存 (Retention)
  └── 复试季内 7日留存 → 目标：40%

付费 (Revenue)
  └── 购买复试包 → 转化率目标：10% → 500付费用户

传播 (Referral)
  └── 分享使用截图/成功案例 → NPS目标：>50
```

### 9.2.1 转化率基准来源说明

> ⚠️ **以上转化率目标均为参照行业数据设定的假设值，需在 Phase 1 上线后用真实数据校准。**

| 漏斗环节 | KaoBot 目标值 | 行业参照基准 | 来源 | 差异说明 |
|---------|-------------|------------|------|---------|
| 曝光→注册（5%）| 5% | 2-8%（垂直工具类 App）| Mixpanel 2024 SaaS Benchmark | KaoBot 场景精准，目标偏上限 |
| 注册→激活（60%）| 60% | 40-70%（教育工具类）| Amplitude 2024 Product Benchmark | 激活定义为"完成一次真题查询"，门槛较低 |
| 7日留存（40%）| 40% | 25-45%（教育 App）| AppsFlyer 移动营销报告 2024 | 复试季使用动机强烈，预计高于均值 |

| NPS > 50 | > 50 | 教育 App 行业均值约 35-45 | Bain & Company NPS 研究 | 高 NPS 依赖产品体验达到预期 |

## 10. 未来展望与优化建议

### 10.1 短期优化建议（0-3个月）

| 优先级 | 优化点 | 预期收益 |
|--------|-------|---------|
| P0 | **PageIndex 替换 Tavily**：建立本地知识库，将真题、导师信息结构化入库 | 准确率从不可控 → ≥95%，知识护城河建立 |
| P0 | **简历 PDF 导出**：POC 只有文本建议，Phase 1 必须输出可投递的成品文件 | 完成真正的价值闭环，支撑付费 |
| P1 | **置信度标注**：所有 AI 回答标注信息来源和置信度（高/中/低） | 降低用户对 AI 幻觉的担忧，提升信任 |
| P1 | **用户档案持久化**：保存用户目标院校/导师/分数，避免重复填写 | 降低使用摩擦，提升留存 |
| P2 | **移动端适配**：当前 CLI 只能 PC 使用，移动端是主要使用场景 | 大幅扩大可用用户群 |

### 10.2 中期产品方向（3-12个月）

1. **知识库社区化**：引入用户贡献机制，让历届考生上传真题/经验，建立"考研 Wikipedia"
2. **导师画像深化**：通过 Semantic Scholar API 自动抓取导师近年论文，动态更新研究趋势
3. **模拟面试升级**：引入语音交互（STT+TTS），让面试模拟更接近真实场景，降低临场陌生感
4. **调剂信息实时化**：接入研招网 API / 爬虫，3月调剂季实现分钟级信息推送

### 10.3 长期愿景（1-3年）

```
愿景: 成为考研全周期的 AI 学业助手

备考期（9-12月） → 复试期（2-4月） → 入学后 → 求职期
   KaoBot            KaoBot            KaoBot    KaoBot
   初试规划           复试全程           学术辅导  求职助手
```

**扩展路径**：
- **横向扩展**：从考研 → 考博 → 出国留学 → 职场晋升（同样存在信息不对称 + 高焦虑的场景）
- **纵向扩展**：从复试辅导 → 入学后导师沟通助手 → 科研写作助手（陪伴用户整个研究生生涯）
- **平台化**：开放知识库给高校/培训机构，成为考研信息领域的"数据基础设施"

### 10.4 市场与业务风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 大厂入场（字节/百度做考研 AI） | 中 | 高 | 加速建立数据飞轮，做大厂不愿做的精细化运营 |
| AI 幻觉导致考生决策失误 | 中 | 高 | 所有信息强制来源标注 + 置信度标注，免责声明 |
| 真题版权纠纷 | 低 | 中 | 法律咨询，明确"用于学习研究"的合理使用边界 |
| LLM API 成本上涨 | 低 | 中 | 建立本地模型备用方案（Qwen/DeepSeek 本地部署） |
| 考研政策变化（如取消复试） | 极低 | 极高 | 产品设计保留向其他考试场景迁移的能力 |

### 10.5 技术风险矩阵

| 技术风险 | 概率 | 影响 | 监控指标 | 应急预案 |
|---------|------|------|---------|---------|
| **LLM 幻觉率超标** | 高 | 高 | 每周抽样 100 条回复，人工评估事实准确率，目标 ≥ 95% | 触发条件：准确率 < 90%；预案：对该类问题强制切换到知识库精确匹配，禁用开放生成 |
| **LLM API 服务中断** | 中 | 高 | 主力模型可用性监控（Uptime Robot），P99 可用性目标 99.5% | 主力中断 → 自动切换备用模型（主：DeepSeek → 备：Kimi → 兜底：本地 Qwen）切换时间 < 30 秒 |
| **知识库数据陈旧** | 中 | 高 | 监控知识库最后更新时间；院校复试信息超过 60 天未更新则标记 | 触发告警 → 人工核查 + 补充更新；对用户显示数据更新日期 |
| **并发性能瓶颈** | 中（复试季高峰）| 高 | 复试季期间每日监控 P99 响应时延（目标 < 5s）；并发请求队列长度 | 队列 > 100 时触发限流（排队提示）；提前 2 周完成压测（目标：1,000 并发无超时）|
| **PDF 生成失败率** | 中 | 中 | 简历 PDF 生成成功率，目标 ≥ 99% | 失败自动重试 2 次；仍失败则降级为 HTML 预览并提供截图导出 |
| **数据库主从延迟** | 低 | 中 | MySQL 主从延迟监控（目标 < 1s）| 延迟 > 5s 强制切主库读写；报警通知 On-Call |
| **用户数据泄露** | 极低 | 极高 | 异常访问检测（单 IP 异常大量请求）；敏感接口访问日志 | 立即封禁异常 IP → 数据加密核查 → 72 小时内通知受影响用户（PIPL 要求）|

## 11. 数据埋点规范

> **背景**：PRD 设定了大量量化目标（漏斗转化率、LTV、留存率等），若无对应埋点则上线后无法有效衡量，产品决策将依赖猜测而非数据。本节定义核心功能的埋点事件列表，作为前后端开发的追踪实现依据。

### 11.1 埋点规范说明

```
事件命名规范：{模块}_{对象}_{动作}
  模块：auth / onboard / feature / purchase / ai / system
  对象：user / query / resume / interview / payment 等
  动作：start / complete / fail / click / view / submit 等

示例：ai_query_start / purchase_complete / resume_download_click

上报时机：事件发生时实时上报（非批量）
上报方式：HTTP POST → 埋点服务（可选用神策分析 / Mixpanel / 自建）
用户标识：user_id（登录后）/ anonymous_id（登录前，设备级）
通用字段（每条事件自动附带）：
  · timestamp      事件发生时间（UTC）
  · user_id        用户ID（登录后）
  · anonymous_id   匿名ID（登录前）
  · session_id     会话ID
  · platform       iOS / Android / Web
  · app_version    应用版本号
  · utm_source     来源渠道（首次安装时记录）
```

## 12. MVP 验收标准（Definition of Done）

> **背景**：M1 MVP 里程碑的"成功指标"（如"问答准确率≥95%"）未说明测试数据集、测试场景和验收流程，导致开发完成后无法客观判断是否"真的完成了"。本节为 M1 MVP 建立明确的 DoD。

### 12.1 M1 MVP 功能验收标准（Functional Acceptance Criteria）

#### F1：导师信息查询

```
验收用例集（共 20 个测试用例）：

正常场景（15个）：
  TC-F1-001：查询"清华大学 计算机系 张X" → 返回导师基本信息（姓名/职称/研究方向）
  TC-F1-002：查询"北京大学 物理学院 XXX" → 返回结果含近3年代表性论文
  TC-F1-003：查询不同学科类型（理工/文科/医学）各5个导师 → 均有结果
  TC-F1-004：输入仅姓名（无院校）→ 返回同名导师列表供用户选择
  TC-F1-005：查询招生状态 → 返回"当年是否招生"（含置信度标注）
  ...（完整用例集见附件 test-cases/F1-teacher-query.xlsx）

边界场景（3个）：
  TC-F1-016：查询已退休导师 → 显示"该导师已退休，信息供参考"提示
  TC-F1-017：查询新晋青年教师（数据不足）→ 诚实返回"暂无充足信息"而非编造
  TC-F1-018：中英文姓名混用 → 正确识别同一导师

异常场景（2个）：
  TC-F1-019：网络断开时查询 → 展示网络异常页面（见框架层异常态设计）
  TC-F1-020：知识库数据超过60天未更新 → 回复中标注"数据可能未反映最新情况"

验收通过条件：20个用例中 ≥18 个通过（通过率 ≥ 90%）
```

#### AI 回复准确率（核心质量指标）

```
测试数据集构建方法：
  1. 从知识库中随机抽取 100 个问题（覆盖导师/真题/院校信息三类）
  2. 由 2 名领域专家（考研从业者）独立对 AI 回答进行评分（0-5分）
  3. 取两者平均分，≥ 4 分视为"准确"

准确率计算：准确条数 / 100

分类准确率要求：
  · 客观信息类（导师姓名/院校/招生信息）：准确率 ≥ 95%
  · 分析推断类（导师偏好/复试重点预测）：准确率 ≥ 80%（允许不确定性）
  · 生成类（简历/自我介绍）：内容完整性 ≥ 90%；格式正确率 100%

⚠️ 特别要求：幻觉零容忍场景（以下类型不得出现捏造）：
  · 导师姓名/职称/院校 → 准确率必须 100%（可返回"暂不确定"，但不得捏造）
  · 历年复试题目 → 来源标注准确率必须 100%
  · 录取分数线 → 来源标注准确率必须 100%
```

#### 性能验收标准

```
测试环境：阿里云 ECS 4核8G（等同生产单节点）
测试工具：k6 / JMeter
测试场景：

场景A（正常负载）：50并发用户，持续10分钟
  目标：P50 < 2s，P95 < 5s，P99 < 8s，错误率 < 1%

场景B（复试季高峰模拟）：200并发用户，持续5分钟
  目标：P50 < 4s，P95 < 10s，错误率 < 5%（允许部分降级）

场景C（LLM 接口降级测试）：主力LLM人工关闭，模拟切换
  目标：备用模型切换完成时间 < 30s，用户无感知（toast提示即可）

PDF生成性能：
  单次生成时间 < 15s（P95）
  并发生成（10个同时）成功率 ≥ 99%
```

### 14.2 数据质量标准

```
质量维度与验收标准：

  完整性：院校真题覆盖率 ≥ 85%（目标院校范围内）
          导师信息完整率 ≥ 90%（姓名/方向/联系方式至少三项）

  准确性：AI 基于知识库的客观信息回答准确率 ≥ 95%
          录取线数据与官方研招网误差 ≤ 0
          （官方数据为唯一真实来源，不做推算）

  时效性：真题数据每年考研季（11月）前完成年度更新
          导师信息每学期开学后 30 天内完成更新扫描
          调剂信息更新延迟 ≤ 5 分钟（调剂窗口期 SLA）

  一致性：同一院校/导师信息在系统内唯一，
          不允许重复条目（去重率 = 100%）
```

### 14.3 数据更新流程

| 数据类型 | 触发条件 | 执行人 | SLA | 验收标准 |
|---------|---------|-------|-----|---------|
| 年度真题更新 | 每年 10 月 1 日启动，11 月 15 日完成 | 内容运营 + AI 辅助解析 | 45 天 | 新增院校≥30所，准确率抽查≥95% |
| 导师信息更新 | 每年 3 月、9 月学期初 | 爬虫自动 + 人工校验异常 | 30 天 | 变更率>10%的院校逐一人工确认 |

| 调剂信息 | 调剂系统开放期间全程实时 | 自动爬虫（无人值守）| ≤ 5 分钟延迟 | 监控告警：延迟>10分钟自动告警 |

### 14.4 数据安全策略

```
数据分级：
  L1（公开数据）：院校真题、导师公开信息、录取线
    → 可缓存、可公开展示、无特殊保护
  L2（用户数据）：手机号、邮箱、上传简历、对话记录
    → AES-256 加密存储；对话记录默认 24h 后清除
  L3（商业敏感）：付费记录、CAC/LTV 分析数据
    → 仅 PM/CEO 可访问；独立数据库，无外部接口

数据合规义务：
  · 用户注销时，30 天内删除其全部 L2 数据（个保法 §47）
  · 数据泄露事件：72 小时内报告主管部门（网安法 §42）
  · 定期数据安全审计：每季度一次，由技术负责人执行

数据出境：
  · Phase 1/2 所有数据存储于中国大陆（阿里云华东2）
  · 不向境外传输用户个人信息（个保法 §38）
  · LLM API 调用（DeepSeek/Kimi/Qwen）：均为国内服务商，合规
```

### 14.5 数据备份与灾难恢复

| 备份类型 | 频率 | 保留周期 | 存储位置 | 恢复测试频率 |

## 15. A/B 测试计划

> **优先级：P2** | A/B 测试是驱动产品决策数据化的关键机制，优先在付费转化和核心功能价值验证上投入。

### 15.1 测试框架与工具

```
工具选型：GrowthBook（开源，自部署，免费）
  · 支持特征开关（Feature Flags）+ A/B 实验
  · 与埋点系统（§11）对接，自动拉取实验指标
  · 统计引擎：贝叶斯统计（适合小样本快速决策）

流量分配原则：
  · 仅对已注册用户做分组（避免 SEO 和首次印象污染）
  · 最小样本量：每组 ≥ 100 用户 × 7 天（短期指标）
  · 互斥原则：同一用户同一时间只参与 ≤ 2 个实验
  · 保守原则：Phase 1 流量 < 500 用户时，优先顺序测试而非并行
```

### 15.2 A/B 测试优先级矩阵

| 实验编号 | 实验主题 | 业务目标 | 优先级 | 预计启动时间 | 最小样本/组 |
|---------|---------|---------|-------|------------|-----------|
| AB-01 | 定价页面布局（锚定策略）| 提升付费转化率 | ⭐⭐⭐ P0 | M1 上线后 2 周 | 150 用户 |
| AB-02 | 首页核心 CTA 文案 | 提升注册转化率 | ⭐⭐⭐ P0 | M1 上线后 1 周 | 200 用户 |
| AB-03 | 功能入口优先级（首页布局）| 验证 F3 vs F1 哪个更吸引用户 | ⭐⭐ P1 | M1 上线后 4 周 | 100 用户 |
| AB-04 | 邀请机制奖励设计 | 提升 K 因子 | ⭐⭐ P1 | M2 阶段 | 100 用户 |
| AB-05 | AI 回答格式（结构化卡片 vs 自由文本）| 提升满意度/续问率 | ⭐ P2 | M2 阶段 | 80 用户 |
| AB-06 | 付费套餐命名（功能导向 vs 情感导向）| 提升付费转化率 | ⭐ P2 | M2 阶段 | 120 用户 |

### 15.3 重点实验设计

#### AB-01：定价页面锚定策略

```
假设：在定价页加入"旗舰套餐"（高价锚定）可提升中间套餐选择率

对照组（A）：
  两档定价：复试包 ¥99 | 月会员 ¥29

实验组（B）：
  三档定价（加入高价锚定）：
    调剂应急包 ¥199（锚）| 复试包 ¥99（目标推荐）| 7天试用 ¥19

主指标：复试包付费转化率（7 天窗口）
次指标：人均收入（ARPU）、页面停留时长
成功标准：实验组主指标提升 ≥ 15%，统计显著性 p < 0.05
```

#### AB-02：首页 CTA 文案测试

```
假设：情感化文案（强调焦虑缓解）优于功能化文案（描述工具能力）

对照组（A）："智能查询导师 · 一键生成简历"（功能导向）
实验组（B）："3天搞定复试备考，考上你的理想院校"（情感导向）

主指标：注册转化率（访客→注册）
次指标：跳出率、首次功能使用率
成功标准：注册率提升 ≥ 20%，p < 0.05
```

#### AB-03：功能优先级验证

```
假设：用户进入产品后，最先需要的是"真题/考点"(F3)而非"导师信息"(F1)

对照组（A）：首页功能入口顺序 F1→F3→F4→F5
实验组（B）：首页功能入口顺序 F3→F1→F4→F5

主指标：7 天功能使用深度（人均使用功能数）
次指标：各功能点击率、DAU/MAU
成功标准：实验组人均功能使用数提升 ≥ 10%
```

### 15.4 实验治理规范

```
实验生命周期：
  1. 提案（PM填写实验设计文档）→ 2. 评审（PM+研发确认技术可行性）
  → 3. 实施（研发接入GrowthBook Flag）→ 4. 运行（≥7天，达到最小样本量）
  → 5. 分析（PM出具实验报告）→ 6. 决策（上线/回滚/继续迭代）

实验结果处理原则：
  · 显著正向（p<0.05，主指标提升≥目标值）：全量发布，3个工作日内
  · 无显著差异：维持对照组，归档实验结论
  · 显著负向：立即停止实验，回滚
  · 样本不足：延长实验周期，最长不超过 21 天

历史实验存档路径：docs/experiments/YYYY-MM-AB编号-实验报告.md
```

## 16. 客服与反馈机制

> **优先级：P2** | 高质量的反馈闭环是早期产品快速迭代的核心信息来源，也是建立用户信任的关键。

### 16.1 反馈渠道设计

| 渠道 | 使用场景 | 响应 SLA | 工具 | Phase 开放 |
|-----|---------|---------|------|-----------|
| **产品内反馈按钮** | 每个 AI 回答下方"👍/👎 + 一句话反馈" | 无需响应（自动收集）| 自建 + 埋点 | Phase 1 |
| **邮件客服** | 账号/付费问题 | 1 个工作日 | 企业邮箱（support@kaobot.cn）| Phase 1 |
| **微信群（种子用户专属）**| 深度反馈、Bug 报告、功能建议 | 4 小时（工作时间）| 企业微信群 | Phase 1 |
| **公开反馈看板**（可选）| 展示"用户投票的功能需求"建立透明度 | — | Canny.io 或国内替代品 | Phase 2 |
| **NPS 调查**（季度）| 量化用户忠诚度，定位改进方向 | — | 腾讯问卷 / TypeForm | Phase 1（上线后4周）|

### 16.2 FAQ 维护策略

```
FAQ 内容来源：
  · 种子用户群常见问题（每周归纳 Top 5）
  · 邮件客服重复问题（出现 ≥ 3 次则进 FAQ）
  · AI 回答不准确的反馈（触发知识库补充流程）

FAQ 结构（分层）：
  第一层：新手入门（注册/登录/付费/退款）
  第二层：功能使用（各功能 F1-F6 操作说明）
  第三层：AI 准确性说明（AI 免责声明 + 已知局限性）

维护频率：每两周更新一次；重大产品变更后 24h 内更新
维护责任人：运营（Phase 1 由 PM 兼任）
存放位置：产品内帮助中心（Web）+ docs/faq/ 目录（内部参考版）
```

### 16.3 投诉处理流程

```
投诉分级处理：

  P0 投诉（付款未到账 / 数据泄露 / 法律投诉）：
    → 30 分钟内响应 → 2 小时内给出处理方案 → CEO 知会
    → 处理结果 24 小时内邮件回复用户

  P1 投诉（AI 回答严重错误影响备考 / 功能完全不可用）：
    → 4 小时内响应 → 1 个工作日内修复或降级方案
    → 酌情补偿（延长会员 7 天 / 退款）

  P2 投诉（AI 回答不准确但有参考价值 / 体验问题）：
    → 1 个工作日内响应 → 进入 Bug Backlog
    → 回复用户处理计划

  通用处理原则：
    · 退款政策：付费后 7 天内，AI 回答严重错误，全额退款
    · 不接受"AI 回答不如人工辅导"类投诉（产品定位差异，需在落地页说明）
    · 所有投诉记录归档于 docs/complaints/ 目录，月度分析趋势
```

### 16.4 反馈数据闭环

```
反馈 → 分析 → 行动 循环（每两周一次）：

  数据输入：
    · AI 回答 👎 率（目标 < 5%）
    · 邮件/微信群反馈主题分类（功能 / AI准确性 / 体验 / 付费）
    · NPS 分数（目标 Phase 1 末 ≥ 30，Phase 2 末 ≥ 50）

  分析输出：
    · 反馈主题 Top 3（输入下期 Sprint 需求排期）
    · AI 错误频发类型（输入知识库补充任务）
    · 用户情绪趋势（输入运营策略调整）

  行动标准：
    · 同一问题收到 ≥ 5 次反馈 → 进入 P1 Sprint 待办
    · AI 某类问题准确率 < 90% → 立即启动知识库补充
    · NPS < 20 → PM 启动专项用户访谈（至少 5 人）
```
